The .NET framework has a number of technologies that allow you to create HTTP services such as Web Service, WCF and now Web API.
 Web Service   
  • It is based on SOAP and return data in XML formt
  • It support only HTTP protocol
  • It is not open source but can be consumed by any client that understands XML
  • It can only be hosted on IIS
Issue: Only runs on HTTP and support SOAP
 WCF   
  • It is also based on SOAP and return data in XML form
  • It support various protocols like TCP, HTTP, HTTPS, MSMQ, Named Pipes
  • It is also not open source but can be consumed by any client that understands XML
  • It can be hosted on IIS, Self or WAS
Issue: Required extensive configuration.
 WCF Rest   
  • It support XML, JSON and ATOM data format
  • To use WCF as Rest, you have to enable webHttpBidings
  • It support GET and POST verbs by 'WebGet' and 'WebInvoke' attributes
  • To enable other HTTP verbs, configuration in IIS required
  • The UriTemplate must specified for parameters in WebGet
Issue: To enable Rest again lot of configuration and additional steps required.
 ASP.NET MVC   
We can extend controller to return data in JSOn format and can accept GET, PUT, DELETE and other verbs of HTTP.
This is become difficult because we try to match the name of verb inside action method. Sometime ends with collision same method one for GET, one for PUT etc.
Nee to add every level...but in Web API verbs are directly mapped to method.
 Web API   
New framework for building HTTP service in simple and easy way.
  • Response are formatted by Web API's MediaTypeFormatter in JSON, XML or whatever you want...
  • It is full features of HTTP
  • It is open source and ideal for RESTful services on .NET
  • Can be hosted with in application or IIS
  • Light weight and good for limited bandwidth devices like smart phones
  • It also supports MVC features such as routing, controllers, action results, filter, model binders etc.
Note: Go for WCF if you want to create a service that can use fast transport when available.
